02

Device & Register

Device는 Installation에 설치되는 물리적 계량기입니다. 하나의 Device는 하나 이상의 Register(계량 항목)를 가질 수 있습니다.

Device

물리적 계량기 자체. 일련번호, 제조사, 설치일 등 기술 속성을 가지며 Installation에 설치(Install)되는 시점과 탈거(Remove)되는 시점이 이력으로 관리됩니다.

Register

Device 내에서 실제로 값을 읽어내는 계량 항목. 예: 시간대별 요금제(계시별)를 쓰는 전력 계량기는 최대부하/일반/경부하 등 여러 Register를 하나의 Device 안에 가질 수 있습니다.

일반화 주의 "Device 1개 = Register 1개"로 단순화하지 않습니다. 계시별 요금(시간대별 차등요금)을 적용받는 산업용 고객의 경우 하나의 계량기 안에 여러 Register가 존재하는 것이 일반적입니다.
03

Meter Reading (검침)

검침은 Register로부터 실제 사용량 값을 수집하는 프로세스입니다. 검침 지시(Meter Reading Order)가 생성되고, 그 결과로 검침 결과(Meter Reading Result)가 시스템에 입력됩니다.

수동 검침검침원이 현장 방문 또는 고객 자가 신고로 값을 입력. 검침 주기가 길거나 AMI 미보급 지역에서 사용
AMI(원격 검침)계량기가 통신망을 통해 자동으로 값을 전송. 검침 빈도와 정확도가 크게 향상되며, 상세 사용 패턴(Interval Data) 확보 가능
기억할 포인트 검침 결과는 그 자체로 청구서가 되지 않습니다. 이 값은 Ch.04의 Billing 단계로 넘어가 요금 구조(Rate)와 결합되어야 비로소 금액이 산출됩니다.
04

AMI 연계 — HES / MDMS 흐름

대형 공기업은 대부분 AMI(Advanced Metering Infrastructure) 기반 원격 검침을 확대하고 있습니다. IS-U는 이 인프라와 직접 통신하지 않고, 중간 시스템을 통해 검침 데이터를 전달받는 구조가 일반적입니다.

MDMS수집된 대용량 검침 데이터를 검증·정제(VEE: Validation, Estimation, Editing)하고 활용 가능한 형태로 관리
IS-UMDMS로부터 정제된 검침 결과를 전달받아 Billing에 활용

05

EDM Overview (Interval Data 개념)

AMI가 확대되면 계량기가 하루에도 수십~수백 회씩 값을 전송합니다. 이렇게 짧은 주기로 수집되는 대용량 데이터를 Interval Data라고 하며, 이를 전문적으로 관리하는 영역이 EDM(Energy Data Management)입니다.

왜 EDM이 필요한가

  • 월 1회 수준의 일반 검침과 달리, 15분/1시간 단위의 대용량 데이터 처리 필요
  • 계시별 요금, 수요관리(Demand Response) 등 정교한 요금 설계의 기반 데이터
  • 일반 Meter Reading 구조만으로는 이 볼륨과 활용 목적을 감당하기 어려움

이 챕터에서 다루는 범위

  • Interval Data의 개념
  • AMI → HES → MDMS → EDM → Billing 흐름
  • EDM이 왜 별도 영역으로 분리되는지
심화 과정 안내 Interval Profile, Profile Values, VEE(Validation·Estimation·Editing), Profile Allocation, Settlement, 시장/정산 연계 등 EDM의 상세 내용은 EDM Advanced 과정(별도 모듈)에서 다룹니다. 이 챕터에서는 "왜 필요한가"까지만 이해하면 충분합니다.
06

PM Integration (설비관리 연계)

계량기 교체, 정기 점검, 고장 수리 등은 IS-U만의 프로세스가 아니라 PM(Plant Maintenance) 모듈과 만나는 지점입니다. 대형 공기업 프로젝트에서는 배전설비 점검처럼 IS-U 범위를 벗어나는 설비관리 프로세스가 함께 설계되는 경우가 많습니다.

검증 필요 — 일반화 주의 IS-U의 Connection Object가 PM의 Functional Location과 항상 1:1로 직접 매핑되는 표준 구조라고 단정할 수 없습니다. IS-U의 Business/Technical Master Data와 PM의 Technical Object(Functional Location, Equipment)를 연계하는 것은 대표적인 통합 패턴 중 하나이며, 실제 매핑 방식과 연계 범위는 프로젝트별 설계 결정 사항입니다. 교육에서는 "대표적인 통합 패턴"과 "프로젝트별 설계 차이"를 함께 설명하는 것이 안전합니다.

※ PM 자체의 상세 구조(Functional Location 계층, Work Order 프로세스 등)는 이 시리즈의 범위를 벗어나며, 여기서는 IS-U와 만나는 지점만 개념적으로 다룹니다.

07

S/4HANA Utilities Note

S/4HANA NOTE — Ch.03

AMI·EDM 영역은 S/4HANA 전환 시 아키텍처 변화가 큰 영역 중 하나로 자주 언급됩니다. 다만 구체적인 변화 폭은 채택하는 EDM 솔루션과 프로젝트 범위에 따라 달라지므로, 이 시리즈에서는 "AMI/EDM 연계가 프로젝트마다 상이하게 설계될 수 있다"는 점을 원칙으로 안내하며, 세부 아키텍처 비교는 본 개론 시리즈의 범위를 벗어납니다.

이해도 점검

Q1. Device와 Register의 관계를 가장 정확히 설명한 것은?
하나의 Device는 항상 하나의 Register만 가진다
하나의 Device는 계시별 요금 등의 이유로 여러 Register를 가질 수 있다
Register가 Device를 포함하는 상위 개념이다
Device와 Register는 동일한 객체를 부르는 다른 이름이다
Q2. AMI 검침 데이터가 IS-U에 도달하기까지의 일반적인 흐름은?
스마트 계량기 → IS-U → HES → MDMS
스마트 계량기 → HES → MDMS → IS-U
MDMS → HES → 스마트 계량기 → IS-U
스마트 계량기 → MDMS → IS-U → HES
Q3. IS-U와 PM의 연계에 대한 설명으로 가장 정확한 것은?
Connection Object와 PM Functional Location은 항상 표준적으로 1:1 매핑된다
IS-U와 PM의 연계 패턴은 대표적인 통합 방식이 있지만, 실제 매핑과 범위는 프로젝트별 설계에 따라 달라질 수 있다
IS-U는 PM과 어떠한 경우에도 연계되지 않는다
PM Work Order는 항상 Meter Reading Order를 통해서만 생성된다

Chapter 03 체크리스트

  • Device와 Register의 관계와 계시별 요금과의 연관성을 설명할 수 있다
  • Meter Reading Order → 검침 → Result → Billing 트리거 흐름을 말할 수 있다
  • AMI(스마트 계량기-HES-MDMS-IS-U) 흐름을 순서대로 설명할 수 있다
  • EDM이 왜 필요한지, 이 챕터와 EDM Advanced 과정의 범위 차이를 안다
  • IS-U-PM 연계가 표준 1:1 매핑이 아니라 프로젝트별 설계 영역임을 이해했다
