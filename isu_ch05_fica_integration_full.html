FI-CA Integration

01

청구서 발행 이후, 채권은 어떻게 관리되는가

Ch.04에서 청구 금액이 확정되어 Invoicing까지 완료되면, 그 금액은 FI-CA Posting을 통해 Contract Account 단위의 채권으로 전기됩니다. Ch.01에서 언급했듯, 대형 유틸리티 프로젝트에서는 이 이후의 Customer Financial Processing이 Billing보다 훨씬 오래, 훨씬 복잡하게 이어지는 경우가 많습니다.

Invoicing
(Ch.04)
→
FI-CA
Posting
→
수납 /
체납 관리
→
FI (GL)

이 챕터는 정상 수납부터 최악의 경우(대손)까지, 채권이 거칠 수 있는 경로를 순서대로 다룹니다.

  • 정상 흐름: Payment Lot(수납) → GL Posting
  • 체납 발생 시: Dunning(독촉) → Collections(채권관리) → Write-off(대손)
  • 사전 리스크 관리: Security Deposit(보증금)
  • 고객 배려: Installment Plan(분할납부)
  • 수납 실패: Return(반환/부도)
02

Payment Lot (수납 처리)

Payment Lot은 은행 등에서 들어온 대량의 입금 데이터를 일괄로 처리하여, 각 Contract Account의 미수금과 매칭(Clearing)하는 프로세스입니다.

입금 데이터
수신(은행 등)
→
Payment Lot
생성
→
Contract Account
미수금 매칭
→
Clearing
(채권 소멸)
일반화 주의 모든 입금이 자동으로 매칭되는 것은 아닙니다. 입금자명 불일치, 금액 불일치 등으로 자동 매칭에 실패하는 미확정 입금 건이 발생할 수 있으며, 이 경우 수작업 또는 별도 로직으로 처리됩니다.
03

Dunning (체납 독촉)

납기 내 수납이 되지 않은 채권에 대해, 단계적으로 강도를 높여가며 고객에게 독촉하는 프로세스입니다. 일반적으로 여러 단계(Dunning Level)로 구성되며, 단계가 올라갈수록 조치의 강도가 높아집니다.

단계일반적 조치 예시
1단계안내성 독촉장 발송 (연체 알림)
2단계공식 독촉장 발송, 납부 기한 재통보
3단계 이상단전 예고, 단전 실행 등 강한 조치 (유틸리티 특화)
유틸리티 특화 포인트 일반 기업의 Dunning은 대부분 문서 발송에서 그치지만, 유틸리티에서는 최종 단계에서 공급 중단(단전/단수)이라는 물리적 조치로 이어질 수 있다는 점이 가장 큰 차이입니다. 다만 단전 실행 조건과 절차는 규제·법령의 영향을 크게 받는 영역으로, 프로젝트마다 상이하게 설계됩니다.
04

Collections (채권 관리)

Dunning으로도 해결되지 않는 장기 연체 건을 별도로 그룹핑하여 집중 관리하는 프로세스입니다. 연체 심각도에 따라 케이스를 분류하고, 담당자를 지정해 후속 조치를 추적합니다.

  • 연체 채권의 케이스화(Case) 및 우선순위 분류
  • 담당자 배정 및 조치 이력 추적
  • Write-off(대손 처리) 여부를 최종 판단하는 전 단계
05

Security Deposit (보증금)

신용도가 낮은 신규 고객이나 특정 요건에 해당하는 고객에게 계약 시점에 미리 징수하는 보증금입니다. 향후 체납이 발생했을 때 손실을 완화하는 사전적 리스크 관리 수단입니다.

일반화 주의 "모든 신규 고객에게 보증금을 징수한다"고 일반화하지 않습니다. 보증금 징수 여부와 기준(신용평가 결과, 계약 종류 등)은 회사 정책과 규제 요건에 따라 프로젝트별로 다르게 설계됩니다.
06

Installment Plan (분할납부)

일시 납부가 어려운 고객의 채권을 여러 회차로 나누어 납부할 수 있도록 재구성하는 프로세스입니다.

계산 예시 — 연체 채권 300,000원을 3개월 분할

총 연체액 = 300,000원
분할 회차 = 3회
─────────────────────────
회차별 납부액 = 300,000원 ÷ 3회 = 100,000원 / 회
기억할 포인트 Installment Plan이 수립되면 원래의 단일 채권이 여러 회차의 개별 채권으로 재구성되며, 각 회차는 독립적으로 Dunning 대상이 될 수 있습니다.
07

Return (반환/부도)

자동이체 등으로 이미 정상 처리된 것으로 보였던 수납이, 잔액 부족이나 계좌 문제 등의 사유로 사후에 취소(반환)되는 경우를 처리하는 프로세스입니다.

자동이체
수납 처리
→
Clearing
(채권 소멸 처리)
→
Return 발생
(부도/잔액부족)
→
채권 재생성
(Reopen)
기억할 포인트 Return이 발생하면 이미 소멸 처리했던 채권이 다시 살아나는(Reopen) 구조이므로, Return 처리 이전에 이미 진행된 Dunning 이력과의 정합성 관리가 중요한 실무 포인트입니다.
08

Write-off (대손 처리)

Collections 단계를 거쳐도 회수가 불가능하다고 최종 판단된 채권을 장부에서 대손으로 처리하는 프로세스입니다. 이는 IS-U/FI-CA만의 절차가 아니라, 최종적으로 FI(GL)의 대손 회계처리와 연결됩니다.

일반화 주의 Write-off의 구체적 승인 기준(연체 기간, 금액 한도 등)과 회계처리 방식은 회사의 대손 정책 및 K-IFRS 등 적용 회계기준에 따라 결정되는 영역으로, 이 시리즈에서 특정 기준을 일반화하여 제시하지 않습니다.
09

GL Posting — FI-CA에서 FI로

지금까지 다룬 모든 FI-CA 프로세스(청구, 수납, 체납, 분할납부, 대손 등)는 개별 Contract Account 단위로 상세하게 기록되지만, 최종적으로는 요약된 형태로 총계정원장(FI GL)에 반영됩니다.

FI-CA
(CA 단위 상세)
→
요약/집계
→
FI
(GL 계정)
Ch.01 재확인 "IS-U는 FI-AR을 아예 쓰지 않는다"가 아니라, FI-CA가 FI-AR의 역할을 대신하며 최종적으로 FI에 반영된다는 Ch.01의 원칙이 여기서 완성됩니다. FI-CA는 상세 채권 관리를, FI는 재무제표 수준의 요약을 담당하는 역할 분담 구조입니다.
10

S/4HANA Utilities Note

S/4HANA NOTE — Ch.05

FI-CA on S/4HANA에서 자주 언급되는 특징은 Universal Journal을 통한 재무 데이터 통합 관리 강화입니다. FI-CA의 프로세스 자체(Payment Lot, Dunning, Collections 등)는 개념적으로 Classic IS-U와 동일하게 유지되며, 주로 데이터 모델과 리포팅 측면에서의 변화가 언급됩니다. 세부 기능 차이는 프로젝트의 S/4HANA 버전에 따라 달라질 수 있어 별도 확인이 필요합니다.

이해도 점검

Q1. 연체 채권 300,000원을 3개월 분할납부로 재구성할 때, 회차별 납부액은?
300,000원 (전액 1회 납부)
100,000원
150,000원
50,000원
Q2. Dunning에서 유틸리티 산업의 가장 큰 특징은 무엇입니까?
문서 발송이 전혀 이루어지지 않는다
Dunning 단계가 항상 1단계로 끝난다
최종 단계에서 공급 중단(단전/단수)이라는 물리적 조치로 이어질 수 있다
Dunning은 IS-U에서 사용되지 않는 개념이다
Q3. Return(반환/부도) 프로세스에 대한 설명으로 옳은 것은?
Return이 발생해도 이미 소멸된 채권은 다시 생성되지 않는다
Return이 발생하면 이미 Clearing 처리됐던 채권이 다시 열린다(Reopen)
Return은 Payment Lot과 무관한 별도의 독립 프로세스다
Return은 항상 Write-off로 직결된다

Chapter 05 체크리스트

  • Payment Lot을 통한 수납·Clearing 흐름을 설명할 수 있다
  • Dunning의 단계적 구조와 유틸리티 특화 조치(단전)를 이해했다
  • Collections와 Write-off의 관계(장기 연체 → 최종 대손 판단)를 안다
  • Security Deposit과 Installment Plan의 목적 차이를 설명할 수 있다
  • Return 발생 시 채권이 Reopen된다는 점을 이해했다
  • FI-CA가 상세 관리, FI가 요약 반영을 담당하는 역할 분담 구조를 설명할 수 있다
