01

이 챕터가 시리즈 전체의 기초인 이유

Ch.01에서 본 E2E 흐름의 앞부분 — Business Partner → Contract Account → Contract → Connection Object → Premise → Installation — 은 단순한 순서가 아니라 서로 다른 성격의 객체가 계층적으로 연결된 구조입니다. 이 구조를 정확히 이해하지 못하면 Ch.04의 요금체계(Rate)를 이해할 수 없습니다.

핵심 전제 Rate(요금)는 Installation에 할당됩니다. 즉, "누가(BP) 어떤 계약(Contract)으로 어느 지점(Installation)에 어떤 요금을 적용받는가"라는 질문에 답하려면, 이 6개 객체의 역할과 관계를 먼저 알아야 합니다.

이 챕터는 크게 두 축으로 나눠 이해합니다.

재무·계약 축 (Business Master Data)

Business Partner → Contract Account → Contract
"누가, 어떤 조건으로 계약했는가"

물리적 위치 축 (Technical Master Data)

Connection Object → Premise → Installation
"어디에, 어떤 형태로 공급되는가"

이 두 축은 Contract에서 만납니다. Contract는 재무·계약 축(BP/CA)과 물리적 위치 축(Installation)을 연결하는 다리 역할을 합니다.

02

Business Partner (BP)

Business Partner는 개인 또는 법인 등 IS-U와 관계를 맺는 모든 대상의 마스터 데이터입니다. 고객뿐 아니라 협력업체, 직원 등도 BP로 관리될 수 있으며, IS-U만이 아니라 CRM 등 다른 SAP 컴포넌트와도 공유되는 범용 마스터 데이터 객체입니다.

항목설명
역할고객(또는 관계 당사자)의 신원 정보 — 성명/상호, 주소, 연락처, 신분 정보 등
범위개인(Person), 조직(Organization), 그룹(Group) 등 BP Category로 구분
IS-U와의 관계BP 자체는 계약이나 재무 정보를 담지 않으며, Contract Account를 통해 재무 프로세스와 연결됨
일반화 주의 "1 BP = 1 고객 = 1 계약"으로 단순화하지 않습니다. 하나의 BP가 여러 Contract Account를 가질 수 있고(예: 주거용/사업용 분리 관리), 반대로 여러 BP가 하나의 Contract Account를 공유하는 구조(예: 공동명의)도 프로젝트 설계에 따라 가능합니다.
03

Contract Account (CA)

Contract Account는 재무 처리의 단위입니다. Ch.01에서 다룬 FI-CA가 실제로 작동하는 대상이 바로 CA이며, 청구서 발행·수금·체납 관리·Dunning이 모두 CA 단위로 그룹핑되어 처리됩니다.

CA가 담당하는 것

  • 청구서 통합 발행 단위
  • 수납·미수금 관리 단위
  • Dunning(체납 독촉) 처리 단위
  • 분할납부, 이체 정보 등 결제 조건

CA가 담당하지 않는 것

  • 실제 공급 서비스 내용 (→ Contract)
  • 공급 지점의 물리적 위치 (→ Installation)
  • 계량·검침 데이터 (→ Device/Register)
설계 포인트 하나의 BP가 여러 CA를 가질 수 있다는 점은, 실무적으로 "한 고객이 여러 공급 지점의 청구를 분리해서 받고 싶어할 때" 유용합니다. 예를 들어 ASPN Energy Korea의 산업용 고객이 본사와 공장의 청구서를 별도로 받고 싶다면, CA를 분리하는 설계가 검토됩니다.
04

Contract

Contract는 BP/CA(재무·계약 축)와 Installation(물리적 위치 축)을 연결하는 객체로, "이 CA로 관리되는 이 BP가 이 Installation에서 이 서비스(예: 전력)를 공급받는다"는 실제 공급 계약 관계를 나타냅니다.

Contract는 재무 축(BP＋CA)과 물리적 축(Installation)이 만나는 연결점입니다.

Contract는 일반적으로 특정 Division(사업부문 — 예: 전력, 가스)에 속하며, 하나의 Installation에 대해 Division별로 별도의 Contract가 존재할 수 있습니다.

05

물리적 위치 계층: Connection Object → Premise → Installation

이 세 객체는 "어디에 공급되는가"를 나타내는 기술 마스터 데이터(Technical Master Data)입니다. 상위에서 하위로 갈수록 더 구체적인 단위가 됩니다.

Connection Object

건물 등 물리적 구조물 단위. 하나의 건물에 여러 Premise가 속할 수 있는 상위 컨테이너 역할.

Premise

Connection Object 내에서 물리적으로 구분되는 공간 단위(예: 특정 호실/구역). 주소 정보를 가짐.

Installation

Premise 내에서 실제 공급 서비스(Division)를 계량·요금 처리하는 논리적 단위. Rate가 할당되는 지점.

CO Connection Object — 예: ASPN Energy Korea 공급 대상 건물
PR Premise — 예: 건물 내 특정 호실/구역
IN Installation — 예: 해당 호실의 전력 공급 (Division: 전력)
DV Device / Register — Ch.03에서 상세
일반화 주의 Connection Object : Premise : Installation의 관계를 항상 "1:1:1"로 단순화하면 안 됩니다. 하나의 Connection Object에 여러 Premise가, 하나의 Premise에도 Division별로 여러 Installation이 존재할 수 있는 1:N 구조가 표준적인 설계 여지입니다.

S/4HANA NOTE — Ch.02

Business Partner는 S/4HANA 전반에서 Customer/Vendor Master를 대체하는 통합 마스터 데이터 모델(BP 중심 설계)의 일부로, IS-U의 BP 구조는 이 흐름과 방향이 일치합니다. Contract Account~Installation 계층 자체는 Classic IS-U와 구조적으로 동일하게 유지됩니다.
