이커머스 ERP — 특징 및 구축 고려사항
이커머스 ERP는 실시간 주문·재고·정산의 통합 처리가 핵심이며, 월마감 컷오프 설계와 대용량 FI 전송 아키텍처를 구축 초기에 확정해야 함
※ Amazon 사례: 공개 IR·AWS 블로그·Seller Central 문서 기반 재구성. 내부 아키텍처 세부사항은 비공개이며 추정·해석 포함.
Section 01
이커머스 ERP 프로세스 특징
일반 제조·유통 ERP와의 차별점 — 5대 핵심 영역
01 · 주문관리 (OMS)
다채널 주문 실시간 통합
자사몰·오픈마켓·SNS커머스 등 다채널 주문을 단일 OMS에서 실시간 통합하고 자동 분배. 채널별 주문 상태 동기화 및 취소·변경 자동 처리 필수. 일반 ERP와 달리 영업사원 없이 주문이 24시간 비정형적으로 발생.
Amazon 사례
Seller Central API로 마켓플레이스·자사 채널 주문을 단일 OMS에 통합. 주문·재고·고객 데이터를 실시간 양방향 동기화.
02 · 재고관리
SKU 단위 실시간 차감 및 과판매 방지
SKU 단위 실시간 재고 차감과 과판매(Overselling) 방지 로직 필수. 멀티 창고·멀티 채널 재고 할당 정책을 사전 정의해야 하며, 채널 간 동일 재고 이중 판매 차단 설계가 핵심.
Amazon 사례
FBA: SKU·주문·물리적 이동을 near-real-time 추적. AI 수요 예측과 자동 예외처리로 과판매 원천 차단. ML 기반 자동 보충(Replenishment) 내재화.
03 · 물류·배송 (WMS)
역물류(반품) 프로세스 통합
WMS·택배사 API 실시간 연동. 반품·교환 역물류(Reverse Logistics) 프로세스를 ERP 재고·회계와 통합 설계. 단계별로 재고 계정·손익 계정이 달라지므로 프로세스 정의 후 ERP 매핑 필수.
Amazon 사례
반품 클릭 → 자동 레이블 생성 → ASIN·주문ID 매칭 → 검수·재입고 또는 Warehouse Deals 분류까지 완전 자동화. FBA 기준 접수 후 3–5 영업일 처리 완료.
04 · 정산·수수료
플랫폼별 정산 주기 및 채널 매출 분리
플랫폼별 수수료 차감, PG사 정산 주기 관리(플랫폼마다 상이), 채널별 매출 분리 인식. 미정산 채권 Accrual 기준 및 쿠폰 할인의 매출 차감 vs. 판촉비 처리 기준을 회계 정책으로 사전 확정 필요.
Amazon 사례
Seller Central에서 채널별 수수료·환불액·SAFE-T 클레임을 이유 코드(Reason Code) 기반으로 분리 추적. 정책 적합 반품은 2일 내 자동 환불.
05 · 고객·CRM
회원 등급·포인트·쿠폰 ERP 연동
회원 등급·포인트·쿠폰을 ERP 매출 인식과 연동. 쿠폰 할인의 매출 차감 vs. 판촉비 처리 기준을 회계 정책으로 확정해야 하며, 구매 이력 기반 CS 처리와 ERP 이력 연동도 설계 범위에 포함.
Amazon 사례
ML 기반 동적 가격 책정·맞춤 추천이 핵심 워크플로우에 내재화. 별도 Add-on이 아닌 Core 설계로 구현.

구축 시 핵심 고려사항
5대 Design Point — #02·#03은 Section 03에서 심화 다룸
01 · 플랫폼 연동 아키텍처
REST API 표준화 및 어댑터 구조

Amazon 사례
결제·재고·배송을 독립 배포·확장 가능한 모듈로 분리. 서비스별 독립 장애 격리(Fault Isolation) 구현으로 단일 장애 영향 최소화.

04 · 반품·역물류 자동화
접수→검수→재입고→환불 단계 자동화
반품 단계별 재고·회계 처리 자동화. 상품 상태 판정(재입고 가능/Warehouse Deals/폐기) 기준을 시스템에 사전 정의해야 하며, 각 분기마다 FI 전표 처리 방식이 다름.
Amazon 사례
AI·자동화 소팅으로 반품 분류. 상태별 자동 분기 처리. 예측 분석으로 반품 패턴 사전 감지 및 재고 최적화.
05 · 확장성 및 피크 대응
시즌 트래픽 대비 수평 확장 아키텍처
블랙프라이데이·설날 등 이벤트 트래픽 급증 대비. 피크 시에도 FI Staging 데이터 정합성이 유지되도록 Queue 기반 비동기 처리 설계 권장.
Amazon 사례
마이크로서비스 전환 후 주문 처리 용량 300% 향상. Prime Day 기준 초당 66,000+ 트랜잭션. 과거 데이터 기반 예측형(Predictive) 사전 스케일링 적용.
Section 03 · 심화
데이터 처리 아키텍처 및 월마감 · FI 결산 설계
이커머스 특유의 복잡성이 집중되는 영역 — 구축 초기에 회계·IT·운영 3자가 공동 확정해야 함
Layer 1 실시간 vs. 배치 처리 분리 원칙
처리 유형 대상 데이터 방식 비고
실시간 주문 접수, 재고 차감, 결제 승인 API Event-driven ms 단위 처리 — FI 직접 전기 불가
준실시간 배송 상태 갱신, 반품 접수 단주기 배치 (15–30분) WMS 연동 주기와 동기화
야간 배치 채널별 매출 집계, PG 정산 대사, FI 전표 생성 일 1회 Batch 월마감 전일 특별 배치 별도 설계 필요
설계 원칙: 재고·주문은 실시간 처리하되 FI 직접 전기는 지양. 집계 후 Staging Table 경유하여 야간 배치로 FI 전기하는 구조가 성능·정합성 양립에 유리.
Amazon 사례
주문 처리 트랜잭션 600건/초 실시간 처리. 재고·장바구니는 DynamoDB로 ms 단위 갱신. Prime Day 기준 초당 66,000+ 트랜잭션을 서비스별 독립 스케일아웃으로 대응.
Layer 2 대용량 데이터 → FI 전송 설계
핵심 과제: 일 수십만~수백만 건 주문 트랜잭션을 어떻게 FI로 넘길 것인가. 집계 레벨과 인터페이스 방식을 구축 초기에 확정해야 하며, 이후 변경 시 전면 재설계가 필요함.
설계 요소 옵션 A — 단건 전송 옵션 B — 집계 후 전송 권장
집계 레벨 주문 1건 = FI 전표 1건 일별 × 채널 × 매출유형 × 결제수단 합산 → FI 전표 1건
장점 개별 추적성 높음 FI 부하 최소화, 월마감 속도 향상
단점 FI 과부하, 결산 지연 위험 원천 데이터는 OMS/DW에서 별도 보관 필요
인터페이스 API Real-time Posting Staging Table → 야간 배치 RFC/IDOC
집계 기준 설계 예시: 채널 × 매출유형(정상매출 / 반품 / 쿠폰할인) × 결제수단 × 일자 — 이 기준이 FI 계정 과목 매핑과 1:1 대응되도록 설계해야 이후 드릴다운 및 감사 대응이 가능함.
Amazon 사례
Seller Central 리포트에서 이유 코드(Reason Code) 기반 반품 분류로 회계 처리 자동화. 정산 주기 불일치분은 미수채권으로 자동 계상하여 FI 전기 지연 최소화.
Layer 3 월마감 컷오프 및 FI 결산 개시 설계
이커머스 특유의 난점: 월말 이후에도 반품·취소·PG 정산이 지속 발생. 마감 기준일 확정과 FI 결산 개시 시점 단축이 이커머스 ERP의 핵심 운영 과제.
① 마감 컷오프(Cut-off) 설계 — 3대 영역
  • 주문·매출: 월말 23:59 기준 확정 — 배송 완료 기준 vs. 결제 승인 기준 중 사전 정책 결정 필수. 기준에 따라 당월 매출 규모가 달라짐.
  • PG 정산: 플랫폼별 정산 주기 상이 (예: 익월 15일 정산, 주 단위 정산 등) → 미정산분을 월말 Accrual로 인식하는 기준 및 회계 처리 방식 설계 필요.
  • 반품: 마감 후 접수분 → 익월 처리 vs. 당월 소급 적용 — 중요성(Materiality) 기준으로 회계 정책 확정 후 시스템에 반영.
